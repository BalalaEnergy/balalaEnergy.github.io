工厂方法

工厂方法 Factory Method

定义一个用于创建对象的接口，让子类决定实例化哪一个类。Factory Method使一个类的实例化延迟到其子类。

工厂方法可以隐藏对象的创建细节。比如Integer.valueOf()方法,使用方大多数情况下只需要Integer的值，至于这个值是从缓存中获取的，还是实例化得到的，并不重要。但使用缓存可以节省一些时间。

public class Integer {
    //...
    public static Integer valueOf(int i) {
        if (i >= IntegerCache.low && i <= IntegerCache.high)
            return IntegerCache.cache[i + (-IntegerCache.low)];
        return new Integer(i);
    }
    //...
}

通常情况下，我们定义工厂方法时，需要提供一个工厂接口和一些对应的实现类。

public class Car {

}

public class Suv extends Car {

}

public interface CarFactory {
    private static final CarFactory factory;

    public CarFactory(CarFactory factory) {
        this.factory = factory;
    }

    public static CarFactory getFactory() {
        return factory;
    }

    Car create();
}

public class SuvFactoryImpl implements CarFactory {
    @java.lang.Override
    public Car create() {
        return new Suv();
    }
}

上方的例子中使用方只需要实例化CarFactory时传入对应的实现类便可以创建出各种不同类型的车辆。

如果应用场景很简单，我们只需要提供一个static方法用于创建该对象即可。这种方式称为静态工厂方法 static factory method。Java标准库中很多地方都用到了这种方法。

public class Demo {
    public static void main(String[] args) {
        Integer.valueOf(100);
        List.of("1", "2", "3");
        LocalDateTime.now();
        //很多，不一一列举了....
    }
}

public interface List<E> extends Collection<E> {
    static <E> List<E> of() {
        return ImmutableCollections.emptyList();
    }
}

其中，List.of() 是写在接口中的。调用方获取的产品总是List接口，而且并不关心它的实际类型。即使调用方知道List产品的实际类型是java.util.ImmutableCollections$ListN，也不要去强制转型为子类，因为静态工厂方法List.of() 保证返回List，但也完全可以修改为返回java.util.ArrayList。这就是里氏替换原则：返回实现接口的任意子类都可以满足该方法的要求，且不影响调用方。

总是引用接口而非实现类，能允许变换子类而不影响调用方，即尽可能面向抽象编程。
